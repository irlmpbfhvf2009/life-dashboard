# 成本控制（Cost Control）

目標：在維持「真實、可部署」的前提下，讓 Life Dashboard 以 **$0**（或盡量接近）
運作。這份文件說明哪些是免費的、哪些「可能」產生費用、以及如何避免。

---

## 一句話總結

| 服務 | 使用方案 | 作品集展示的典型費用 |
| --- | --- | --- |
| Vercel | Hobby | **免費** |
| Google Cloud Run | 按用量計費，可縮放到 **0** | **約免費**（在免費額度內） |
| Neon PostgreSQL | Free 方案 | **免費** |
| Firebase Authentication | Spark（免費） | **免費** |

不用 Cloud SQL、不用付費 Redis、不用付費 AI API、不用常駐實例。

---

## 1. Google Cloud Run — 最需要留意的地方

Cloud Run 是在**處理請求期間**才對 CPU／記憶體計費（再加上少量的請求次數費）。
依照我們的設定：

- `--min-instances 0` → **閒置時沒有任何實例在跑 = 閒置不計費。** 這是最重要的設定，
  展示用途**絕對不要**把它設成大於 0。
- `--max-instances 1` → 限制你可能被計費的上限，並保護 Neon 的連線數。
- `--memory 512Mi --cpu 1` → 佔用很小。只有在出現 OOM（記憶體不足）時才調到 `1Gi`。
- `--concurrency 40` → 一個實例同時服務多個請求，所以很少需要啟動第二個。

**免費額度（每月，實際以 Google 當前條款為準）：** 約 200 萬次請求、
36 萬 GB-秒的記憶體、18 萬 vCPU-秒。一個個人儀表板遠遠用不到這麼多。

**哪些情況可能產生費用：**
- 把 `min-instances >= 1`（讓 JVM 24 小時都暖機著）→ **這會產生費用。** 請避免。
- 客戶端不斷狂打 API 的迴圈／失控請求 → `max-instances 1` 加上 Cloud Run 的請求逾時
  （`--timeout 60`）可以把影響範圍限制住。
- 對外網路流量對 JSON API 來說通常可忽略不計。

**防護措施：**
- 設定**預算警示**：帳單 → 預算與快訊 → 建立預算（例如 $1），在 50/90/100% 時寄信通知。
- 冷啟動是「縮放到 0」的代價。對作品集來說可以接受。

## 2. Neon PostgreSQL

Free 方案包含相當充足的儲存空間與運算時數，而且會**縮放到 0**
（閒置時資料庫會自動暫停，有連線時再喚醒）。

**哪些情況可能產生費用／碰到上限：**
- 在高用量下超過儲存或運算時數的額度。展示用途不會碰到。
- **連線數：** 免費方案的連線數有限。我們用很小的 Hikari 連線池
  （`maximum-pool-size: 3`、`minimum-idle: 0`）加上 `max-instances 1` 來緩解，
  並使用 Neon 的 **pooled** 連線主機。

**防護措施：**
- 保持 `min-instances 0`，這樣就不會一直佔著閒置的 DB 連線。
- 使用連線池端點（主機含 `-pooler`）。

---

## 3. Vercel（Hobby）

前端是一個靜態 SPA 打包檔 → 在 **Hobby** 方案上由 Vercel 的 CDN 免費提供
（非商業用途）。頻寬／建置額度對展示用途來說綽綽有餘。

**哪些情況可能產生費用：** 商業用途需要 Pro 方案 — 不在本專案範圍內。

---

## 4. Firebase Authentication

Google 登入在 **Spark（免費）**方案下，一般用量沒有每次認證的費用。我們**只**用
Authentication — 沒有用 Firestore、Storage、Functions 或 Hosting — 所以沒有其他
會計費的東西。

**防護措施：**
- 除非你刻意要加入付費的 Firebase 產品，否則不要啟用帳單／升級到 Blaze。

---

## 5. 我們刻意避開的東西（為了維持免費）

- ❌ **Cloud SQL** — 常駐、按小時計費。改用 Neon（可縮放到 0）。
- ❌ **付費 Redis / Memorystore** — 不需要；沒有快取層。
- ❌ **付費 AI API** — 不在範圍內。
- ❌ **常駐後端** — 改用 Cloud Run 縮放到 0。

---

## 6. 部署前檢查清單

- [ ] Cloud Run `min-instances = 0`、`max-instances = 1`。
- [ ] 已建立 GCP **預算警示**（例如 $1 門檻）。
- [ ] Neon 使用 **pooled** 主機；Hikari 連線池很小（已設定好）。
- [ ] Firebase 仍維持 **Spark**（沒有升級 Blaze）。
- [ ] 沒有 commit 任何機密；全部透過環境變數／Secret Manager。
- [ ] Vercel 專案維持 **Hobby**。

---

## 7. 如果之後需要「保持暖機」

冷啟動讓你困擾嗎？以下選項由便宜到貴排列：
1. 接受冷啟動（免費）。
2. 用輕量的外部 uptime 服務每幾分鐘 ping 一次 — 能減少感受到的冷啟動，但**有可能**
   產生足夠的呼叫次數／CPU 而超出免費額度，記得盯著預算警示。
3. `min-instances 1` — 穩定但**不免費**（一個 JVM 會持續計費）。

對作品集來說，建議選第 1 個。
